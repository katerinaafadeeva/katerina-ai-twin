## 1. Глоссарий

| Термин | Определение |
|--------|-------------|
| **Вакансия** | Запись о позиции из HH.ru или TG-канала, сохранённая в `job_raw` |
| **Скоринг** | LLM-оценка вакансии по 10-балльной шкале (Claude Haiku) |
| **Отклик** | Фактическая заявка на вакансию, отправленная через HH.ru (Playwright) |
| **Сопроводительное** | Cover letter, сгенерированный LLM для конкретной вакансии |
| **Pipeline** | Цепочка: Ingest → Score → Policy → Cover Letter → Apply → Notify |
| **HH-вакансия** | Вакансия, полученная автоматически через API HH.ru (`source=hh_api`) |
| **TG-вакансия** | Вакансия, пересланная пользователем из TG-канала (`source=telegram_forward`) |
| **Apply Cap** | Дневной лимит реальных откликов. **Главный ограничитель pipeline.** |
| **Scoring Cap** | Safety net на LLM-расходы. Не должен мешать набору apply cap. |
| **AUTO_APPLY** | Автоматический отклик без одобрения пользователя (score 5-6, source=hh) |
| **AUTO_QUEUE** | Постановка в очередь без автоматического отклика (score 5-6, source=telegram_forward) |
| **APPROVAL_REQUIRED** | Требуется одобрение пользователя перед откликом (score ≥ 7) |
| **IGNORE** | Вакансия не подходит (score < 5) |
| **HOLD** | Отклик отложен (дневной apply лимит исчерпан) |

---

## 2. Цели и метрики

### 2.1. Бизнес-цель

Автоматизировать поиск работы: бот самостоятельно находит вакансии, оценивает их, генерирует сопроводительные письма и отправляет отклики на HH.ru, формируя **реальную воронку откликов** с минимальным участием пользователя.

### 2.2. Ключевые метрики MVP v1

| Метрика | Target | Описание |
|---------|--------|----------|
| Реальные отклики Вт-Чт | ≥ 40 | Фактически отправленные на HH.ru с сопроводительным |
| Реальные отклики Пн | ≥ 20 | 50% от пикового |
| Реальные отклики Пт | ≥ 8 | 20% от пикового |
| Реальные отклики Сб/Вс | 0 | Заблокировано |
| Время approve → отклик | ≤ 90 сек | После нажатия «Одобрить» |
| TG-форвард → скоринг | ≤ 2 мин | После пересылки |
| % откликов с cover letter | 100% | Без исключений |
| Грамматический род | Женский | «полезна», «готова», «уверена» |

### 2.3. Принцип ограничителей

**Apply Cap — главный и единственный ограничитель воронки.** Бот должен скорить столько вакансий, сколько нужно, чтобы набрать apply cap. Scoring cap существует только как safety net на LLM-расходы (защита от аномального расхода), он НЕ должен мешать набору целевого количества откликов.

Пропущенные вакансии (IGNORE, skipped, already_applied) НЕ считаются в apply cap. Только фактически отправленные отклики.

## 3. Функциональные требования

### 3.1. Источники вакансий

#### FR-SRC-01: HH.ru API Ingest

Бот автоматически загружает вакансии через анонимный API HH.ru по заданным поисковым запросам.

- Частота: каждые 30 минут
- Дедупликация: по `hh_vacancy_id`
- Хранение: `job_raw` с `source='hh_api'`

#### FR-SRC-02: TG Forward

Пользователь пересылает сообщение из TG-канала → бот извлекает текст и сохраняет.

- Ответ пользователю: «✅ Сохранено: #ID» + статус
- Формат ответа:
  - Если TG-кап свободен: `✅ Сохранено: #1500 ⏳ Оценка через ~1-2 мин (2/20)`
  - Если TG-кап исчерпан: `✅ Сохранено: #1500 ⏸ TG-лимит исчерпан (20/20). Оценка завтра.`

#### FR-SRC-03: Умное извлечение ссылок из TG-сообщений

Если в пересланном TG-сообщении есть ссылки (любого формата — не только HH), бот ДОЛЖЕН:

1. Извлечь все URL из сообщения
2. Определить какая ссылка ведёт к полному описанию вакансии. Критерии: ключевые слова рядом со ссылкой («описание», «подробнее», «подробности», «откликнуться», «вакансия»), или домен HH.ru.
3. Перейти по ссылке (HTTP fetch) и извлечь текст описания
4. Использовать полное описание для скоринга (не только текст TG-поста)
5. Если ссылка на HH → сохранить `hh_vacancy_id` для дедупликации и возможности автоматического отклика

**Определение релевантной ссылки:** можно через regex + ключевые слова (без LLM) или через Haiku с минимальным промптом (дешевле 0.001$). Предпочтительно regex.

**Приоритет:** P0 (необходимо для качественного скоринга TG-вакансий и для UI-слоя Итерации 2).

### 3.2. Скоринг

#### FR-SCORE-01: LLM-оценка

Каждая вакансия оценивается Claude Haiku по шкале 1-10 на основе: релевантность опыту, совпадение по роли и навыкам, формат работы, финансовые условия.

#### FR-SCORE-02: Scoring Cap как safety net

| Параметр | Default | Описание |
|----------|---------|----------|
| `HH_SCORING_DAILY_CAP` | 500 | Safety net: максимум LLM-вызовов для HH в день |
| `TG_SCORING_DAILY_CAP` | 50 | Safety net: максимум LLM-вызовов для TG в день |

**Правила:**
- Лимиты НЕЗАВИСИМЫ: исчерпание HH-капа НЕ блокирует TG-скоринг
- TG-вакансии скорятся ПЕРВЫМИ
- Scoring cap = safety net, а не ограничитель pipeline. В нормальном режиме бот должен скорить достаточно, чтобы набрать apply cap.
- При исчерпании cap — одно уведомление в день

#### FR-SCORE-03: Кэш score по hh_vacancy_id

Если вакансия с тем же `hh_vacancy_id` уже была оценена (через другой `job_raw_id`) — повторный LLM-вызов НЕ нужен. Копируем score из существующей оценки.

### 3.3. Policy Engine

#### FR-POL-01: Таблица решений

| Score | Source | Решение |
|-------|--------|---------|
| < 5 | любой | IGNORE |
| 5-6 | hh_api | AUTO_APPLY |
| 5-6 | telegram_forward | AUTO_QUEUE |
| ≥ 7 | любой | APPROVAL_REQUIRED |
| любой | apply cap исчерпан | HOLD |

#### FR-POL-02: Одна action на вакансию

Для каждого `job_raw_id` — максимум одна action. UNIQUE constraint на `(job_raw_id, action_type)`.

#### FR-POL-03: Дедупликация по hh_vacancy_id

Если вакансия с тем же `hh_vacancy_id` уже имеет успешный отклик — новая action НЕ создаётся. Пропущенная вакансия НЕ считается в apply cap (это не потраченный слот).

### 3.4. Cover Letter

#### FR-CL-01: Генерация для каждого подходящего решения

Cover letter генерируется для AUTO_APPLY, AUTO_QUEUE и APPROVAL_REQUIRED.

#### FR-CL-02: Формат

- Русский, женский род
- 500-800 символов
- Без шаблонных фраз, конкретика

#### FR-CL-03: Обязательное отображение в TG

Текст cover letter ВСЕГДА показывается в уведомлении.

### 3.5. Apply

#### FR-APPLY-01: Автоматический отклик

Для AUTO_APPLY(pending) и APPROVAL_REQUIRED(approved) — Playwright отправляет отклик + cover letter.

#### FR-APPLY-02: Apply Cap — главный ограничитель

| День | Cap | Multiplier |
|------|-----|------------|
| Пн | 20 | 50% |
| Вт-Чт | 40 | 100% |
| Пт | 8 | 20% |
| Сб/Вс | 0 | 0% |

Base: `APPLY_DAILY_CAP=40`. Считаются ТОЛЬКО фактически отправленные отклики. IGNORE, skipped, already_applied — НЕ считаются.

#### FR-APPLY-03: Cover letter обязателен

Каждый отклик содержит сопроводительное. Отклик без письма = брак.

#### FR-APPLY-04: Anti-ban

- Задержка 30-90 сек между откликами
- Batch ≤ 5
- Только Пн-Пт 09:00-20:00 MSK

#### FR-APPLY-05: Проверка «уже откликались»

1. БД: по `hh_vacancy_id`
2. Playwright: индикатор на странице
3. Skipped не считается в apply cap

#### FR-APPLY-06: Approve → немедленный Apply

Нажатие «Одобрить» → apply в течение 90 сек (asyncio.create_task, не ждать цикл worker).

#### FR-APPLY-07: Background apply worker

Автоматический фоновый цикл каждые 2-5 минут. Берёт AUTO_APPLY(pending) + APPROVAL_REQUIRED(approved).

### 3.6. Уведомления

#### FR-NOTIF-01: Одно уведомление на вакансию

AUTO_APPLY → только от apply worker (после отклика).
AUTO_QUEUE → от scoring worker (с cover letter).
APPROVAL_REQUIRED → от scoring worker (с cover letter + inline кнопки).

#### FR-NOTIF-02: Форматы

**После отклика (AUTO_APPLY или approved):**
```
✅ Отклик отправлен
📋 Product Manager — Компания XYZ
🎯 Score: 7/10
🔗 hh.ru/vacancy/12345
📝 Сопроводительное:
[текст]
```

**APPROVAL_REQUIRED:**
```
🟢 Требует одобрения
📋 Project Manager — AYA Group
🎯 Score: 9/10
🔗 hh.ru/vacancy/12345
📝 Сопроводительное:
[текст]
[✅ Одобрить] [❌ Отклонить] [⏸ Отложить]
```

Кнопки — inline keyboard в ТОМ ЖЕ сообщении.

**Batch с названиями:**
```
📋 Batch отклики: ✅2 отправлено
  ✅ Product Manager — Компания A (hh.ru/vacancy/111)
  ✅ PM — Компания B (hh.ru/vacancy/222)
  ⏩ 1 уже откликались:
    • PM — Компания C (hh.ru/vacancy/333)
```

#### FR-NOTIF-03: Cap-уведомления — одно в день per source

### 3.8. LLM-оптимизация

#### FR-LLM-01: Prompt Caching

`cache_control: {"type": "ephemeral"}` на system prompt + резюме. Экономия ~90% input tokens.

#### FR-LLM-02: Pre-filter без LLM

Regex/keyword фильтр ДО LLM-вызова. Отсекает явный мусор: стажёр, intern, junior (без PM/Product), нерелевантные города (если задан фильтр). Экономия 30-50% LLM-вызовов.

#### FR-LLM-03: Score кэш по hh_vacancy_id

Если вакансия с таким `hh_vacancy_id` уже оценена — не скорить повторно.

#### FR-LLM-04: Batch API (P2, Итерация 2)

Anthropic Message Batches API: пачка запросов со скидкой 50%. Для ночного/пакетного скоринга.

#### FR-LLM-05: Модели

| Задача | Модель |
|--------|--------|
| Скоринг | Claude Haiku |
| Cover Letter | Claude Haiku |
| Архитектура | Claude Opus (через Katerina) |
| Реализация | Claude Sonnet (TL в Claude Code) |

## 4. Нефункциональные требования

### NFR-01: Anti-ban
Задержки 30-90 сек, batch ≤ 5, рабочие часы MSK, weekday multiplier.

### NFR-02: Стоимость
LLM ≤ $10/месяц. Инфраструктура $0 (локально).

### NFR-03: Расписание
Scoring: круглосуточно. Apply: Пн-Пт 09:00-20:00 MSK. Все времена — MSK (UTC+3).

## 5. User Stories

### US-01: Автоматическая воронка
**Как** пользователь, **я хочу** чтобы к вечеру на HH было 20-40 новых откликов без моего участия.

AC: ≥40 в Вт-Чт, каждый с cover letter (женский род), нет дублей.

### US-02: TG-пересылка
**Как** пользователь, **я хочу** пересылать вакансии из TG-каналов и получать оценку.

AC: ответ в 1 сек, скоринг ≤ 2 мин, независимо от HH-капа, если есть ссылка — бот переходит и берёт полное описание.

### US-03: Одобрение
**Как** пользователь, **я хочу** контролировать отклики на score ≥ 7.

AC: кнопки в том же сообщении, approve → отклик ≤ 90 сек.

### US-04: Прозрачность
**Как** пользователь, **я хочу** видеть что происходит.

AC: `/today` с прогресс-барами, `/queue` с названиями, batch с деталями.

## 6. Acceptance Criteria для закрытия MVP v1

- [ ] В Вт/Ср/Чт отправлено ≥ 40 реальных откликов с cover letter
- [ ] Approve → отклик ≤ 90 сек
- [ ] TG-форвард → скоринг независимо от HH, ≤ 2 мин
- [ ] TG-ссылки: бот переходит и извлекает полное описание
- [ ] Нет дублей, нет мусора в очереди
- [ ] Batch-уведомления с названиями и ссылками
- [ ] Prompt caching + pre-filter активны
- [ ] Все тесты зелёные (510+)
- [ ] Код в main, ветки влиты, репо очищен
